2026年3月12日这一天，最核心的推进是把前一日离线 replay 中筛出来的 `num_history` 候选，真正放回 Habitat 闭环里做了一轮完整验证。这个步骤非常关键，因为昨天我们已经看到，把历史帧数量从 8 降到 4、2、0，在离线纯推理 benchmark 里可以明显降低 System 2 的时延，看起来像是一个很有希望的轻量优化点。但真正决定这个方向值不值得继续投入的，不是离线速度，而是它回到闭环之后，功能指标是否还能保持稳定。

今天的结果说明，离线和闭环之间存在明显差异。原始 baseline，也就是 `num_history=8` 时，这组 mini 闭环的 Success 达到 1.0，SPL 约为 0.911，Navigation Error 约为 0.916，整体表现是稳定的。而当我们把 `num_history` 降到 4 时，虽然 System 2 的单次平均推理时间从大约 0.513 秒下降到了 0.359 秒，看上去模型层面的确更快了，但闭环功能指标却明显退化：Success 降到了 0.75，SPL 降到了 0.671，Navigation Error 上升到了 2.16。更重要的是，整体每步的 wall-clock 时间并没有像离线 benchmark 那样同步下降，反而略微上升到了 0.534 秒。这说明离线纯推理里看到的速度收益，并没有直接转化成闭环端到端收益。

进一步把 `num_history` 压到 2、1、0 以后，结果就更明确了。虽然 System 2 的推理时间继续下降，但闭环功能几乎完全崩掉，Success 全部降到 0，Navigation Error 持续恶化，而且模型进入 `pixel_goal -> System 1` 的比例也快速下降，到 `num_history=1` 和 `0` 时已经只剩下 4% 左右。换句话说，这已经不是简单的“略微降精度换速度”，而是模型的高层决策行为本身发生了变化，它越来越不愿意把控制权交给 System 1，而是更倾向于直接走离散动作。这一点与我们最初担心的 realworld 问题恰好形成了呼应，也就是一旦上下文不足，大小脑切换会明显减少。

因此，今天最重要的结论其实不是找到了一个可以直接上线的 `num_history` 候选，而是明确排除了一个看起来便宜、但实际上闭环代价过高的优化方向。昨天我们一度觉得，`num_history=4` 也许会是一个既明显提速又不怎么伤功能的低风险方案；但今天的闭环结果表明，这个判断并不成立。它在离线 benchmark 里是有价值的，但不能直接作为部署配置使用。这一步排除是有意义的，因为它帮助我们避免在一个方向上继续投入更多时间。

从整体优化策略上看，这组结果反而让下一阶段更清楚了。既然“减少上下文”这条路虽然能加速，但会明显破坏闭环能力，那么接下来更值得投入的方向，就应该是那些尽量保留原始上下文、但降低推理计算成本的办法，也就是 KV cache、前缀复用以及更高性能的推理后端。这类方法的共同特点是，它们的目标不是删掉视觉历史信息，而是在不改变输入语义的前提下，尽量减少 System 2 每一步重复做的计算。今天的闭环结果实际上强化了这条路线的必要性，因为它说明我们真正需要的是“保留上下文、降低成本”，而不是“牺牲上下文、换取速度”。

因此，3月12日完成的是整个 DualVLN 优化方案中的“闭环验证与方向收敛”这一环。昨天我们得到的是离线 replay 层面的速度结论，而今天我们用 Habitat 闭环把这些结论真正验证了一遍，并确认哪些方向只是离线看起来有效、但闭环里并不成立。这个阶段结束以后，下一步就不应该再继续围绕 `num_history=4/2/1/0` 做更多尝试，而应该正式转入 KV cache、推理后端等更底层的优化实验，同时保持 `num_history=8` 作为功能不退化的参考输入配置。这样推进，能保证后面的优化更接近真正可落地的方案，而不是停留在离线 benchmark 的局部最优上。

今天还顺手解决了一个很现实的工程问题，也就是日志体积。当前 `/root/backup/InternNav/logs` 大约有 70GB，其中将近 68GB 都来自刚刚做完的闭环 `num_history` sweep。进一步拆开以后发现，这些空间几乎都不是结果汇总文件本身，而是每组实验导出的 replay 子集，也就是逐步保存下来的图像、深度和上下文资产。换句话说，我们真正需要长期保留用于汇报和分析的，其实只是 `result.json`、`runtime_summary_rank0.json`、`comparison_num_history_closed_loop.csv` 这一类 summary 文件，而不是那些几十 GB 的逐帧回放资产。因此，今天已经把日志清理也工程化了：新增了一个 dry-run/执行两用的清理脚本，可以安全删除闭环 sweep 中的 `replay_subset`、`replay_subset_v2` 和检查图像目录，在保留结论文件的同时，一次性回收大约 67GB 空间。

更重要的是，为了避免以后再重复踩这个坑，今天也顺手把闭环 sweep runner 做了一个小修正：如果不需要导出 replay，只要把参数设成 `--replay-num-episodes 0`，它现在就会自动关闭 replay 导出，而不会再默认把大量逐帧观测全部落盘。这意味着以后做纯闭环功能对比时，磁盘占用会比今天这次小很多，后续实验执行成本也会更可控。

在优化方向上，今天还把 KV cache 这条路线进一步拆成了可执行的实验分层。当前代码虽然在 `generate()` 内部打开了 `use_cache=True`，但在 evaluator、policy、replay benchmark 和 realworld agent 这些关键入口里，实际上仍然显式传入 `past_key_values=None`。这意味着现在真正生效的只是“单次生成内部的 cache”，而不是跨环境步或跨决策步的 cache 复用。也正因为如此，今天以后更合理的重点，不是再去削弱历史上下文，而是开始研究如何在保留 `num_history=8` 的前提下，减少每一步都重新做 instruction、历史图像和当前图像 prefill 的重复计算。换句话说，方向已经从“删上下文换速度”正式转向“保留上下文、减少重复计算”。

在这个判断基础上，今天也把 KV cache 的前两个实验阶段真正做成了可以执行的代码。第一阶段是“输入压力定位”，也就是在离线 replay benchmark 里，不再只看每一步花了多少时间，而是同步记录这一步到底喂给了模型多少文本 token、多少历史图像、多少视觉 token，以及 prompt 本身有多长。这样做的意义在于，我们后面如果看到 System 2 很慢，就不再只是知道“慢”，而是能区分慢在文本 prefill、慢在多图视觉 prefill，还是慢在真正的生成解码。这一步虽然不直接带来加速，但它能把后续优化方向从猜测变成证据驱动。

第二阶段则是在 realworld agent 这条链路里，加了一个默认关闭的、非常保守的 look-down KV cache 实验开关。这个实验只在 `look_down=True` 的 follow-up 上尝试复用上一轮生成留下来的 cache，不会动正常的主路径，也不会默认改变部署行为。如果新一轮输入与上一轮输出在 token 前缀上不能严格对齐，或者中间任何一步出现图像切片和推理异常，它都会自动退回原始的全量 prefill 逻辑。换句话说，这不是一项“默认启用的优化”，而是一个带完整回退机制的实验分支，目的是先回答两个问题：第一，当前 look-down continuation 到底有多少比例具备 cache 复用条件；第二，就算具备条件，它的收益是否足够大，值得进一步工程化。

为了让这件事可量化，今天同时把 HTTP loopback benchmark 也接上了这个实验开关。后面同一份 `replay_subset_v2` 数据，现在可以直接分别跑一轮“cache 关闭”和“cache 实验模式”，然后对比服务端模型耗时、总 step 耗时，以及最关键的几个命中率指标，比如 look-down continuation 总尝试次数、真正命中 cache 的次数、因为 token 前缀不一致而失败的次数、以及因为异常而回退的次数。这样即使这轮实验最后证明当前 chat template 还不适合直接复用 cache，我们也能明确知道瓶颈到底卡在什么地方，而不是停留在抽象猜测上。

因此，3月12日新增完成的其实是整个优化链路中的“KV cache Phase A / B 实验化”这一环。和前半天做的 `num_history` 闭环 sweep 不同，这一步不是为了立刻拿到新的部署配置，而是为了把后续底层优化的评估方法搭起来。接下来更合适的执行顺序，是先在离线 replay 上跑一轮新的 Phase A baseline，确认当前 System 2 的主要成本是否真的是 prefill；然后再用 HTTP loopback 跑一轮 `disabled` 对比 `lookdown_experimental`，看看 look-down follow-up 这条路径有没有值得继续做的 cache 命中率和时延收益。如果这两步结果积极，后面可以继续往 prefix reuse 或更完整的 KV cache 方向推进；如果结果表明命中率很低、前缀不一致很多，那就应该尽早转向推理后端这类替代路线。这样推进，会比一上来就大改主链路安全得多，也更容易跟老板解释每一步的人力投入为什么值得。

今天还顺手定位了一个很容易误判的问题，也就是为什么有时候明明重新执行了 Habitat mini 闭环，但程序几乎瞬间结束，而且新的 `replay_subset_v2` 并没有生成。原因不是 replay_v2 导出逻辑失效，而是当前评估代码带有 resume 机制：如果输出目录里已经存在 `progress.json`，并且里面记录的那 8 个 episode 都已经完成，那么再次用同一个 output 目录启动时，系统会直接把这些 episode 视为已完成并全部跳过，于是进度条就会显示 `0it`。从表面看像是“没跑起来”，实际上是“被当成已经跑完了”。为了解决这个问题，同时避免覆盖之前已经分析过的 baseline 结果，今天补了一个专门的新 config，把重新生成 replay_v2 的输出目录单独切到了 `test_dual_system_mini_replay_v2`。这样后面再做 HTTP loopback 的 KV cache 对比时，就可以直接基于这个新的 replay_v2 目录继续往下走，不会和前面已有的结果混在一起，也不需要手动删除旧日志。

基于这个新的 replay_v2 目录，今天还完成了 HTTP loopback 的第一轮 baseline，也就是在不启用任何 KV cache 复用的前提下，把当前真实的 client 编码、server 解码、agent 推理这条链整体测了一遍。这个 baseline 的意义，不是看 cache 有没有收益，而是先确认“哪一段最慢、值不值得做 cache”。结果非常有价值。整体上，这组 8 个 episode、645 个 step 的 loopback 基线里，总 step 的中位数大约只有 0.26 秒，但 95 分位却接近 2 秒，说明这条链路不是均匀地慢，而是存在很明显的长尾。进一步拆开以后发现，这个长尾几乎完全来自 look-down continuation，也就是模型在需要向下看以后继续决策的那部分路径。普通非 look-down step 的服务端模型耗时中位数只有大约 0.20 秒，而 look-down 相关 step 的服务端模型耗时中位数已经接近 1.93 秒，基本慢了一个数量级。

这件事的重要性在于，它给当前 KV cache 方案提供了非常直接的证据支持。我们之前把 Phase B 的第一条实验路径选成 `look_down=True` follow-up，更多是基于代码结构和经验判断；而今天这份 loopback baseline 说明，这个选择在数据上也是正确的。当前 loopback 的主要长尾，本来就集中在这条路径上。换句话说，如果后面 KV cache 真能在这个分支上成功命中并减少重复 prefill，那么它最有可能消掉的，正是现在这条链最不经济的那部分时延。反过来说，如果后面实验发现这条路径的 cache 命中率很低，或者 token 前缀经常对不上，那我们也能很快知道，这条路线虽然理论上切中了长尾，但在现有模板和对话组织方式下未必真正可行。

因此，今天这一步的阶段性结论可以概括成一句话：HTTP loopback 的 baseline 已经证明，当前最值得优先优化的不是所有 step，而是 look-down continuation 这条明显的慢路径。这使得下一步的任务也更清楚了。接下来最重要的，不是再去猜 KV cache 应不应该做，而是直接跑一轮 `disabled` 对 `lookdown_experimental` 的 AB 对比，看看这条路径到底能不能真正命中 cache、把服务端模型耗时降下来，同时又不明显破坏输出一致性。如果这一步成立，就值得继续往 prefix reuse 或更完整的 cache 方向深入；如果不成立，也能尽早收缩人力，转向更高性能的推理后端。

今天这组 AB 对比也已经真正跑完了，结论比单纯“有收益/没收益”更具体。实验模式下，look-down continuation 这 93 次机会里，程序实际上 93 次都成功识别出了“可以尝试复用 cache”的场景，也就是说，前缀对齐、增量 token 和最后一张图像的切分逻辑本身是成立的；但在真正调用底层的多模态 `generate(..., past_key_values=...)` 时，这 93 次全部发生异常，随后又都自动退回了原始的全量 prefill 路径。因此最终表现出来的现象是：实验模式和 baseline 的时延、一致性几乎完全一样，看上去像是“开了 cache 也没有收益”，但更准确的说法应该是“当前这条多模态 cache 接口根本没有成功跑通，所以收益没有真正发生”。

这一步其实很重要，因为它帮助我们把问题从“高层思路是否正确”进一步缩小到了“底层接口是否可行”。现在已经基本可以排除 replay 数据不够、look-down 上下文没对齐、或者增量图像切分错误这些高层问题。相反，真正的阻塞点更像是 Hugging Face 这一层的多模态 `generate` 接口本身不接受当前这种带 `past_key_values` 的续接方式，或者在 vision token、位置编码和 cache 位置管理上存在不兼容。换句话说，今天这轮实验的价值，不在于拿到了 cache 的加速，而在于它明确证明：如果继续沿着“直接在现有多模态 generate 接口上续 cache”这条路硬推，风险已经明显上升了。

因此，今天之后的判断也进一步收敛了。KV cache 这条路线本身没有被否定，但当前这条最直接的实现方式，已经显示出较强的接口层阻塞。下一步更合理的做法，不是继续盲目在同一接法上反复试，而是先补足异常日志，把具体失败位置确认下来。如果确认确实是底层 API 的限制，那么就应该尽快把主力转向另外两条更稳的路线：一条是 prefix reuse，也就是在不改整体输入语义的情况下减少前缀 prefill 的重复计算；另一条则是更高性能的 System 2 推理后端。这样做的意义是，我们已经用一次实验把一条看似简单、但实际上可能被框架接口卡死的路线提前识别出来，从而避免后面继续在同一个点上消耗过多人力。

为了让下一步排查更高效，今天还顺手把这条链路的异常调试能力补齐了。之前 loopback benchmark 只能告诉我们“发生了多少次 cache exception”，但看不到到底是哪一种异常、出现在什么 step、当时输入规模有多大。现在这些信息都已经会被直接落盘，包括异常类型、异常消息、traceback、完整输入 token 数、增量 token 数、完整视觉 token 数和增量视觉 token 数。与此同时，还新增了一个“遇到首个异常就停止”的调试脚本。这个脚本会复用现在的 HTTP loopback 路径，但不再跑完整轮 645 个 step，而是在遇到第一条真实 cache 异常时就立刻退出，并把对应的 episode、step 和完整异常栈单独写出来。这样做的目的很直接，就是把后面的问题从“猜是不是框架限制”收敛成“拿到第一条具体栈，然后判断是不是 vision token、cache position，还是 Hugging Face 的多模态 generate 接口本身有结构性限制”。这一步完成后，后面的技术决策就会更有依据，也能更快决定是否要继续投入 KV cache，还是尽早把资源转向 prefix reuse 或更快后端。

今天这条首错 traceback 已经拿到了，而且它把问题进一步收缩到了非常具体的一层。第一，当前在带 `past_key_values` 的 continuation 路径上，如果不显式传 `cache_position`，Hugging Face 会自动推导出一个空的 `cache_position`，随后直接触发索引越界。这说明我们之前的实验分支连最基本的位置索引都还没有喂完整，这一层已经先补上了。第二，也是更关键的一点，即使把 `cache_position` 显式补齐，Qwen2.5-VL 在 continuation 路径里会主动把新的 `pixel_values` 置空。换句话说，它在标准的 `generate + cache` 接口里，本来就不自然支持“继续缓存的同时，再追加一张新的图像输入”这种用法。而这恰恰就是我们当前 look-down continuation 想做的事情。

另外，这次调试脚本在终端里只打印了少量几次 `output` 就结束，表面上看起来像是“没有异常”。但这其实也是符合脚本设计的。这个 debug runner 不是为了像普通程序那样把异常直接打到终端，而是为了在捕获到首个 cache exception 时，马上把异常信息、对应的 episode/step 和输入规模写进一个独立 json，然后主动结束。也正因为如此，终端上看不到长 traceback，并不表示没有异常；相反，真正的异常已经被更完整地保存到了 json 里。这一点也解释了为什么终端里只看到 4 次 output，而 json 里只处理了 2 个 replay step：那 4 次 output 实际上包含了 1 次 warmup、1 次普通 step、1 次下一步的普通调用，以及 1 次 look-down follow-up 调用，并不代表真正跑了 4 个 replay step。

基于这些结果，今天也把下一条更稳的路线定下来了，也就是 prefix reuse。这里的核心判断是：我们现在不建议继续在 `past_key_values + multimodal generate` 这条接法上反复试，不是因为它理论上绝对不可能成功，而是因为它已经暴露出明显的接口层冲突。当前这条路线要求模型在 continuation 阶段一边复用过去的 cache，一边再接收一张新的 look-down 图像；但 Qwen2.5-VL 在标准 continuation 路径里会主动把新的 `pixel_values` 清掉，这说明它本身的设计假设更接近“图像先看完，后面只继续文本”，而不是“继续文本的同时再插入新的图像观测”。如果继续沿这条路硬推，后面需要投入的工作就不再是普通实验，而是要去改更底层的多模态 generate 行为，包括位置索引、视觉 token 注入和 cache 续接逻辑，工程风险会明显上升。

相比之下，prefix reuse 的思路更稳。它不是试图在 continuation 阶段硬塞新的图像，而是先把每一步都重复出现的静态 instruction 和提示模板前缀单独识别出来，评估这些前缀在总多模态输入里到底占了多少比例。如果这部分本来就占得很小，那这条路线不值得继续投入；如果占比不低，那么后面再考虑做真正的 runtime prefix reuse 就更有把握。因此，今天也顺手把这条路线的最小实验版补出来了。新增脚本会直接利用 `replay_subset_v2` 的真实决策上下文，统计每个 S2 决策里总输入 token、图像 token、静态文本前缀 token 以及前缀占比，从而告诉我们 prefix reuse 在理论上最多能带来多大收益。也就是说，接下来我们的目标不是立刻把 prefix reuse 做成生产优化，而是先用数据判断：这条路线值不值得进入下一阶段工程化。

这个最小实验版在第一次跑的时候，还顺带暴露出一个 replay 数据层面的问题：部分决策记录里引用的历史图像路径，在磁盘上并没有真正落盘。如果不处理，这类缺口会让分析脚本直接失败。考虑到这条分析链的目的不是逐帧可视化，而是判断 prefix reuse 的理论空间，因此今天也把脚本做了容错处理：遇到个别缺失历史图时，不再整轮中断，而是跳过缺失图继续分析，并把这类缺口的数量单独记在结果里。这样后续看 prefix reuse 分析结果时，一方面能继续拿到整体趋势，另一方面也能知道这轮数据里有多少历史图是缺失的，避免在数据质量不透明的情况下误解分析结论。

后面继续试跑时，又进一步发现，如果某条决策记录缺的不是“可有可无的一张图”，而是 prompt 里明确占了一个图像占位符的位置，那么就不能简单拿剩下的图继续喂 processor，否则图像数量和文本里的占位符数量对不上，模型预处理阶段还是会直接报错。因此今天把容错逻辑又收紧了一步：只要某条决策记录缺任意一张应有的历史图，就整条记录跳过，不再尝试用残缺输入做分析。与此同时，被跳过的记录也会在 details 文件里明确标记原因。这样虽然会让可分析样本数略微减少，但能保证最后产出的 prefix reuse 分析结果是稳定可信的，而不是建立在“图像数量与 prompt 结构不一致”的错误输入之上。

这个发现非常重要，因为它意味着当前的阻塞点已经不是高层实验设计的问题，也不是 replay 数据不够、前缀对不齐、或者 look-down 图像没切好。真正的问题已经收敛到了底层多模态接口本身：当前这条 Hugging Face/Qwen2.5-VL 的标准 cache continuation 路线，很可能从设计上就不适合我们现在这种“文本续接 + 新图像输入”的场景。也正因为如此，下一步的投入方向也应该更谨慎。我们仍然会把修正后的首错 debug 再跑一轮，把补上 `cache_position` 之后的第一条真实异常完整确认下来；但如果后面继续证明 vision input 在 continuation 里被结构性屏蔽，那么就不应该再在这条接口上硬耗人力，而是尽快把工作重心转向两条更稳的路线：一种是 prefix reuse，也就是尽量减少静态前缀的重复 prefill；另一种是更快的 System 2 推理后端。这样做的意义是，我们现在已经不仅知道“cache 这条实验没跑通”，而且开始清楚“它为什么没跑通”，这会显著减少后续走弯路的概率。

今天也把 prefix reuse 这条替代路线的理论空间真正量化了一次。做法不是直接改模型，而是基于 `replay_v2` 中真实保存下来的决策上下文，统计每次 System 2 决策时，总输入 token 里到底有多少是重复的静态 instruction/template 前缀。最终共有 167 条决策记录进入分析流程，其中 145 条记录成功完成了完整的多模态 processor 对齐；另外有 22 条记录因为历史图像缺失而被跳过，对应缺失图片 32 张。这个数据质量缺口本身已经被单独记录下来，因此不会混入后面的统计结论里。

结果比预期更清楚地说明了问题所在。从纯文本 prompt 的角度看，静态前缀的重复度非常高，平均大约占文本 token 的 71.5%，也就是说如果只看文字，这条路线看起来很有吸引力。但一旦放回真实的多模态输入里，情况就完全不同了：静态前缀平均只占总输入 token 的 3.86%，而图像 token 平均达到 13428.8，远高于文本 token 的 145.5。换句话说，当前 System 2 prefill 的主要成本并不在 instruction 文本前缀，而在视觉输入本身。进一步拆开看，普通决策步里静态前缀约占总输入的 4.56%，而真正更慢的 look-down follow-up 路径里，这个比例反而更低，只有 2.67%。这说明越是当前真正需要优化的慢路径，单纯依靠文本前缀复用能拿到的收益越有限。

因此，3月12日这一步的阶段性结论也进一步明确了。prefix reuse 仍然值得保留，但更适合当作低风险的辅助优化，而不是当前的主攻方向。真正更有希望带来明显端到端收益的，还是两条更重但更直接的路线：一条是更高性能的 System 2 推理后端，另一条是比“纯文本前缀复用”更广义的 partial prefill 或多模态 prefill 优化。这样安排的好处是，团队现在已经不仅知道某些路线“理论上可能快”，而且开始能用实际输入统计判断“哪条路线的理论上限更值得投入人力”，这会让后续优化决策更稳，也更容易向老板解释为什么某些方向应该继续、某些方向应该尽早收缩。

基于这个判断，今天也开始把 vLLM 这条更偏“推理后端”的路线真正落成可执行的实验入口。这里的关键不是简单说一句“可以试 vLLM”，而是先把问题切对。当前这个 DualVLN checkpoint 不是标准的 Qwen2.5-VL 模型，而是带有自定义 `internvla_n1` 架构和一批 System 1 相关模块的模型。如果试图一上来把整套 DualVLN 全部迁到 vLLM，上手复杂度其实会明显增加，也更容易因为模型结构不完全标准而被卡住。因此，今天确定的策略是只把 System 2 这一段拿出来试 vLLM，也就是把它当成一个“图像+指令到文本输出”的视觉语言生成模型来验证，而继续保留现有的 System 1 路径不变。这样做的目的是先回答一个更直接的问题：如果只替换最慢的 System 2 文本生成后端，能不能在不大改整体逻辑的前提下拿到明显收益。

为了让这件事可以马上开始，而不是停留在想法层面，今天已经补了两类脚本。第一类是可行性检查脚本，它会读取当前 checkpoint 的配置和权重索引，判断这个模型离标准 Qwen2.5-VL 还有多远，同时生成一个专门给 S2-only 实验使用的“patched config view”。这个视图不会复制整套权重，而是通过软链接复用原始文件，只重写配置里的 `model_type` 和 `architectures`，把它伪装成标准 Qwen2.5-VL，用来尝试 vLLM 加载。第二类是新的 S2 backend benchmark，它可以在同一份 replay manifest 上分别跑 Hugging Face backend 和 vLLM backend，只比较 System 2 文本生成的冷启动、p50/p95 延时、tokens per second 和输出一致性。也就是说，我们现在已经把“试 vLLM”从一句方向性建议，变成了一个可以直接执行的实验入口。

目前这一步还没有产出新的时延数字，原因不是代码没准备好，而是本地环境里还没有安装 `vllm`。因此，3月12日这部分工作的性质更适合表述为“完成了 vLLM 实验入口与兼容性准备”，而不是“已经验证 vLLM 能带来多少提速”。但这一步本身是有价值的，因为它把下一阶段的后端优化从模糊讨论，推进到了可以直接开始 AB 测试的状态。后面只要安装好 vLLM，就可以直接先跑一轮 S2-only 的 HF vs vLLM replay 对比，再决定这条路线是否值得继续扩大到更完整的系统层面。
