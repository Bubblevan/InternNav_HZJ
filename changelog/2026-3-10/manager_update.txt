2026年3月10日这一天的工作，主要完成的是整个 DualVLN 延迟拆解方案里的第一环，也就是先在 Habitat 仿真中把闭环评估跑通，并把原来只有功能指标的评估流程，扩展成既能看 Success、SPL、Oracle Success、Navigation Error，也能同步看到每个 episode 的总耗时、平均每步耗时、System 2 调用次数、以及 System 2 和 System 1 的行为占比和模块级耗时。这样做的目的是先在一个可控环境里回答我们最关心的问题：真实机器人上明显的高延迟，到底更像是由大脑 System 2 拖慢，还是由小脑 System 1 拖慢，或者是两者之间的切换机制没有充分发生。

今天一个很关键的结论是，当前这条 Habitat 的 VLN-CE R2R 评估链，并不依赖那套体量非常大的 LeRobot `traj_data/r2r`。闭环评估真正需要的只是原始 R2R 标注文件和 Matterport3D 场景资源，因此从实验推进角度看，我们不需要为了做仿真评估额外下载 343GB 的轨迹数据。这一点非常重要，因为它直接消除了最初“磁盘空间不够所以必须找 mini dataset 替代”的前提条件。换句话说，我们之后完全可以在 main branch 的完整数据配置上工作，只是通过限制评估 episode 数量来做 smoke test，而不必被大体积轨迹数据阻塞。

在代码层面，今天已经把 Habitat 评估入口改造成了可分析的版本。除了保留原有的功能评测外，现在每个 episode 都会记录完整的运行时信息，并且可以额外导出一个 replay 子集。这个 replay 子集保存了每一步决策对应的图像、深度、位姿、历史帧索引，以及模型当时的基线输出，后续可以直接拿去做 L20 上的离线纯推理 benchmark，而不需要依赖 Habitat 环境本身。这个改动的意义在于，我们终于把“闭环性能”和“纯模型推理性能”分开了，后面无论是做 KV cache 优化、换推理后端还是做量化，都可以先在离线环境里测到更纯粹的收益，再决定是否放回闭环里验证。

今天还实际跑通了一组 mini R2R 评估样例，总共 8 个 episode。结果表明，这条链路已经能稳定完成闭环，功能指标上 Success 达到 0.875，SPL 大约 0.775，Navigation Error 约为 1.02。从延迟视角看，平均每步 wall-clock 大约是 0.51 秒，System 2 的单次平均耗时约 0.52 秒，而 System 1 的单次平均耗时约 0.153 秒。这说明在模型层面，System 2 的确是更重的部分，它比 System 1 明显更慢。与此同时，我们观察到 System 2 输出离散动作的比例约为 63%，输出 pixel goal 进入 System 1 的比例约为 37%。这个现象和我们之前在 realworld 上的直觉是基本一致的，也就是很多时候 System 2 仍在直接控制动作，没有足够频繁地把控制权交给 System 1 执行连续运动。

但这次仿真也补充了一个更细的判断：问题并不是“System 1 根本没有发生”。相反，在这 8 个 episode 里，System 1 的调用次数和 System 2 已经是同一量级，而且每次进入 System 1 后，平均可以连续执行大约 8.6 步。这说明大小脑切换在仿真中是实际存在的，只是切换比例还不够高。此外，这次 profiling 还告诉我们一个更系统层面的事实：把 System 2 和 System 1 的模型耗时加起来，只占整体 episode wall-clock 的大约 38%。也就是说，剩下超过 60% 的时间并不是模型本身，而是在 Habitat 交互、视角切换、图像深度预处理、以及 Python 侧流程组织里消耗掉了。这个结论对后续优化方向非常关键，因为它意味着即使我们把模型加速了，如果不同时关注交互和系统开销，闭环收益也可能不会等比例体现。

这里也把几个关键指标的含义写清楚。Success 表示最终是否成功抵达目标；SPL 表示不仅成功，而且路径是否足够高效；Oracle Success 表示轨迹中是否曾经到过目标区域；Navigation Error 表示结束时距离目标还有多远，越低越好。这里提到的 wall-clock，可以理解为现实世界时钟真正流逝的时间，而不是单纯模型前向推理的时间。换句话说，episode 的 wall-clock 包含了模型推理、环境交互、视角切换、图像深度预处理和控制逻辑本身，因此它更接近我们在真实机器人上感受到的“到底慢不慢”。

本次 8 个 mini episode 的具体数据如下：Success 为 0.875，SPL 为 0.775，Oracle Success 为 0.875，Navigation Error 为 1.023。平均每个 episode 的总 wall-clock 约为 44.21 秒，平均每一步 wall-clock 约为 0.509 秒。System 2 单次平均耗时约为 0.520 秒，System 1 单次平均耗时约为 0.153 秒。System 2 输出 pixel goal 的比例约为 37.5%，直接输出离散动作的比例约为 62.5%，每次进入 System 1 后平均可连续执行 8.6 步。综合这些数字，可以很明确地说，当前瓶颈首先还是在 System 2，而不是 System 1。

从整体研发节奏上看，3月10日完成的是整个方案里的“仿真闭环打表与离线基线准备”阶段。它的作用不是直接做性能优化，而是先把后续优化所需要的测量工具、重放数据和判断依据建立起来。只有这个阶段完成以后，我们后面对模型做任何单变量改动，才知道到底是让 System 2 更快了，还是只是换了一个指标表面变好但闭环效果退化的实现。

今天还往前推进了一步，不只是把 replay 子集导出来，而且把离线 replay benchmark 脚本也补成了更适合后续 L20 横向对比的结构化版本。后续这个脚本可以分开统计 S2 文本生成、S2 latent 生成、S1 局部轨迹生成以及总推理耗时，并同时输出一致性指标。与此同时，也补写了两份长期有用的说明文档，一份是 DualVLN 在 Habitat 闭环中的代码架构和数据流解释，帮助团队成员更快理解大小脑在代码里是怎样衔接的；另一份是关于 KV cache、推理后端和量化该如何排序推进的实验构思，为后面的性能优化阶段做准备。

在这之后，还实际完成了一轮离线 replay benchmark。这里需要特别说明一下它的结果应该如何理解。这次 benchmark 跑的并不是 1 个 episode，而是 replay 子集中的全部 8 个 episode，总共 697 个 step。运行日志里几乎所有 step 都显示为 discrete，S1 和 latent 都接近 0，乍看会让人误以为离线条件下模型完全没有进入 System 1。但进一步核对 replay manifest 以后发现，真实的 baseline 数据里其实有 496 个 step 属于 pixel goal 类型，只有 201 个 step 是离散动作。这说明问题不在数据本身，而在于当前第一版离线 benchmark 还没有完全复现闭环 evaluator 中的上下文状态。

更准确地说，当前 replay benchmark 是把每一个 step 都独立当成一次新的 S2 输入来测，而闭环里很多 step 实际上只是同一个 pixel-goal 周期中的连续执行步，并不会重新触发一次新的高层决策。同时，当前脚本还没有完整恢复 evaluator 里和多轮上下文、look-down 分支相关的状态，所以它更适合被理解为“纯推理速度 baseline”，而不是“严格等价于闭环过程的大小脑重放”。这个结果依然有价值，因为它非常适合用来做不同参数设置下的延迟横向对比，只是不能简单把离线里大量 `pixel_goal -> discrete` 的失配，直接解释成模型能力退化。

基于这一点，接下来的缩减实验就更清楚了。我们下一轮不应该同时改多个变量，而是优先做三组单变量实验。第一组是 `num_history` 缩减实验，建议从当前的 8 开始，依次测试 4、2 和 0，观察 S2 文本生成延迟和输出一致性是否同步变化，用来判断历史帧长度对 prefill 开销的影响。第二组是 prompt 长度缩减实验，保留一版当前完整 prompt，再设计一版短 prompt 和一版最小 prompt，观察时延下降的同时，输出类型和动作一致性是否明显恶化。第三组是 `max_new_tokens` 缩减实验，建议从 128 依次测试到 64、32、16，重点看生成长度是否真的存在明显冗余，以及缩短输出上限之后，S2 latency 是否下降而功能一致性基本不变。

这三组实验最重要的不是一次性找最优参数，而是先判断性能瓶颈主要落在什么地方。如果 `num_history` 的变化最敏感，就说明当前大头是多图输入带来的 prefill 成本；如果 prompt 缩短最有效，就说明大模型生成过程里仍有较多语言模板冗余；如果 `max_new_tokens` 缩减最有效，则说明当前输出长度配置偏保守，存在直接压缩生成时延的空间。只有在这三组实验明确以后，我们再进入 KV cache 或推理后端优化，才更容易知道优化收益来自哪一部分，而不是在一个不清楚主瓶颈的位置上盲目调参。

为降低后续执行成本，今天还把这三组实验做成了一个统一的 runner。这样后面不需要手动改脚本、改参数、逐个收集结果，而是可以一键生成整组实验和最终对比表。与此同时，也额外整理了一份“严格离线重放”的设计方案。它解决的是另一个更深的问题：如果未来我们不只是想测纯推理速度，而是想尽量在离线条件下严格复现闭环里的大小脑交接，那么现有 replay 数据还缺哪些状态，需要重新跑一次 mini 闭环把哪些决策级上下文保存下来。这个方案已经写成独立文档，后面可以直接作为 replay_v2 的开发蓝图。

下一步建议不要立刻把时间主要投在量化上，而是先把今天导出的 replay 子集用于离线纯推理 benchmark，先把 DualVLN 在 L20 上的冷启动时间、warm latency、tokens per second、显存峰值和输出一致性都测出来。原因在于当前的仿真结果已经很清楚地表明，System 2 是主要模型级瓶颈，而 System 2 的主要负担来自自回归生成过程。对这种瓶颈，通常最值得先试的不是量化，而是 KV cache 相关优化和更高效的推理后端，因为这些方法更直接作用于生成链路本身，也更有机会在不明显损伤输出一致性的情况下拿到收益。

因此，明天更合理的实验顺序应该是先做一轮离线 replay baseline，然后优先测试 KV cache 路径是否已经充分启用，以及是否有必要切换到更高性能的推理后端。等这一步完成以后，再选择一个量化方案作为单一变量来做第二轮离线测试，重点比较延迟、显存和输出一致性。最后，再把收益最好的单变量放回 Habitat 闭环，检查 Success、SPL 等功能指标是否退化。这样推进，能够保证我们不是盲目做优化，而是在一个可解释、可比较、可回退的框架里逐步筛选出最有效的方向。
