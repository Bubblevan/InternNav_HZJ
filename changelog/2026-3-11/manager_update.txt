2026年3月11日的推进，主要是把昨天已经打通的 Habitat 闭环评估链，进一步整理成一套可以直接执行的实验流程，并且把“当前离线 replay 能做什么”和“严格复现闭环还缺什么”这两个问题分开讲清楚。这样做的目的，是避免后续在实验推进中把不同层级的结果混在一起理解，导致判断优化方向时出现偏差。

今天最重要的结论是，当前我们已经有两条可以稳定运行的链路。第一条是 Habitat 闭环评估链，它可以继续输出 Success、SPL、Oracle Success、Navigation Error，以及每个 episode 的 wall-clock 时间、System 2 和 System 1 的调用统计和耗时拆解。第二条是基于 replay 子集的离线纯推理 benchmark 链，它可以在不依赖 Habitat 环境的前提下，测出 DualVLN 在固定输入上的冷启动时间、warm latency、tokens per second、生成长度、显存峰值和输出一致性。也就是说，今天起我们已经可以把“闭环性能”和“纯模型性能”分开看。

同时，今天也把一个关键边界彻底说明白了。当前这一版离线 replay benchmark，适合作为纯推理基线和单变量对比工具，但还不能被理解成严格复现闭环上下文的重放。原因在于，现有 replay 数据记录的是每一个环境 step，而不是每一个真正触发新高层决策的 System 2 节点；此外，多轮上下文、look-down 分支状态以及当时真正输入给模型的精确 prompt 和图像列表，也还没有被完整保存。因此，今天的离线 replay 仍然更适合回答“模型本身在固定输入上有多快”，而不是直接回答“闭环里的大小脑切换是否被一比一复现了”。

基于这一点，今天把后续实验路线明确分成了两个阶段。第一个阶段是马上可以执行的阶段，也就是重新跑一遍 mini 闭环或 smoke 闭环，生成新的 replay 子集，然后立刻在这份 replay 上完成纯推理 baseline 和三组单变量实验。这里的三组实验分别是 `num_history` 缩减、prompt 长度缩减和 `max_new_tokens` 缩减。通过这三组实验，我们可以先判断当前延迟最敏感的是历史图像长度、语言 prompt 冗余，还是生成长度上限配置。这个阶段不需要再新增复杂代码，今天已经可以直接开始执行。

第二个阶段则是更严格的离线重放阶段。为了尽量复现闭环里的大小脑交接，需要重新设计 replay_v2，在导出 replay 时额外保存决策级上下文，例如这个 step 是否真的触发了新的 System 2、高层决策属于哪个 decision id、当前是否处于同一个 pixel-goal 周期中、闭环当时送进模型的精确 prompt 文本、参与决策的图像列表，以及必要时的多轮 conversation 状态。只有把这些状态补全以后，后面才能先做 decision-only replay，再进一步发展到 full-cycle replay。今天已经把这一整套方案写成了可执行的开发顺序，后面可以直接照着推进。

今天还进一步明确了和真实部署的关系。之前我们担心真实机器人上的高延迟，除了模型本身，可能还包含 client/server 通信、图像编码解码、HTTP 往返等额外成本。基于这一点，今天也把“HTTP loopback replay”纳入了整体路线里。它的定位不是替代当前的纯模型 replay，而是作为第三层 fidelity，在未来需要时专门用来估算接近 realworld 部署形态下的通信和序列化开销。这样一来，我们就形成了三层结构：第一层测最纯粹的模型推理速度，第二层测严格上下文下的大小脑复现，第三层再测通信结构带来的系统成本。

从实际执行上看，今天已经把命令级 runbook 整理出来了。现在如果要继续推进，最直接的做法是先重新跑 mini 闭环，或者直接在完整 `val_unseen` 上做 8 个 episode 的 smoke 闭环。闭环结束后，立刻在生成的 replay 子集上运行离线 benchmark 基线，再使用统一 runner 执行三组单变量 sweep，自动产出对比表。这样做的意义在于，我们在进入更复杂的 KV cache、推理后端或量化优化之前，先用最便宜的方式筛出最敏感的变量，避免在错误的瓶颈上投入过多时间。

因此，3月11日这一天完成的是整个 DualVLN 优化方案中的“实验执行流程固化与严格重放方案澄清”这一环。昨天解决的是仿真闭环能否跑通、耗时能否拆开、replay 能否导出；今天解决的是跑通之后应该按什么顺序做实验、哪些结果可以直接用于优化判断、哪些结果暂时还不能被过度解读。这样到明天开始做具体优化实验时，团队已经有了一套比较清楚的基线、命令、结果解释边界和下一步开发方向。

下一步建议很明确。第一步，重新生成一份新的 mini 或 smoke replay。第二步，在这份 replay 上完成纯推理 baseline 和三组缩减实验。第三步，根据结果判断是否先推进 `num_history / prompt / max_new_tokens` 这类轻量优化，还是直接进入 `KV cache / 推理后端` 优化。第四步，并行规划 replay_v2 的字段扩展，为后续更严格地复现大小脑闭环做准备。如果轻量优化已经能带来明显收益，就优先把这些收益放回 Habitat 闭环验证功能指标是否退化；如果轻量优化收益有限，再把主要精力转向更重的推理系统优化。

今天后续又往前推进了一步，原本还停留在方案层的 `replay_v2` 和 HTTP loopback 也已经真正落到代码里了。现在闭环重新运行以后，不只是会生成原版 replay 子集，还会额外生成一份 `replay_subset_v2`。这份新数据比旧版多保存了很多决策级上下文，例如某个 step 是否真的触发了新的 System 2、高层决策属于哪个 `decision_id`、当前是不是同一个 pixel-goal 周期中的延续步、当时真正送进模型的 prompt 文本、输入图像列表和 conversation 状态等。这样后面如果要继续做更严格的离线重放，就不需要再完全依赖猜状态，而是可以直接基于闭环中真实导出的上下文来做。

与此同时，也新增了一条可以直接执行的 HTTP loopback benchmark。它不是单独起一个 server 再让 client 去请求，而是在一个本地进程里模拟 client 侧的图片编码、server 侧的解码和 realworld agent 的 `step(...)` 调用，从而估算接近真实部署时 client/server 结构本身会带来的额外开销。这样到目前为止，我们实际上已经形成了三层离线评测结构：第一层是最纯粹的模型推理 baseline，第二层是保存了决策上下文的 `replay_v2`，第三层则是贴近 realworld 通信封装形态的 loopback 评测。

今天也完成了一轮新的 replay benchmark 结果分析。先看闭环本身，当前这组 8 个 mini episode 的功能指标已经非常稳定，Success 和 Oracle Success 都达到了 1.0，SPL 约为 0.911，Navigation Error 约为 0.916，平均每步 wall-clock 约为 0.508 秒。System 2 单次平均耗时约为 0.513 秒，System 1 单次平均耗时约为 0.153 秒，System 2 输出 pixel goal 的比例约为 37%，而每次进入 System 1 后平均可连续执行约 8.44 步。这个结果继续支持我们此前的判断：在 Habitat 仿真中，大小脑切换是实际发生的，而且模型层面的主要瓶颈仍然是 System 2。

再看离线纯推理 replay 基线。这次 benchmark 一共跑了 8 个 episode、697 个 step。冷启动加载时间约为 3.51 秒，GPU 峰值显存约为 17.5GB，单步总时延的 p50 约为 0.86 秒，p95 约为 0.93 秒。与此同时，System 2 文本生成本身的 p50 和 p95 几乎和总时延完全重合，分别也约为 0.86 秒和 0.93 秒。这说明在当前这条纯推理 benchmark 中，时延几乎完全由 System 2 生成阶段主导。这个结论对后续优化优先级非常关键，因为它进一步强化了“先盯 System 2，而不是先盯 System 1”的判断。

不过，这次 replay benchmark 也再次证明了当前这条 baseline 不能被过度解读。虽然闭环里大约 37% 的高层决策会走 pixel goal 进入 System 1，但这次离线 replay 中真正触发 `s2_latent` 和 `s1_generate` 的只有 24 次；更具体地说，baseline 里本来属于 pixel-goal 的大量 step，在离线重放时都重新落回了 discrete 分支。一致性指标也体现了这一点：输出类型一致率大约只有 29.8%，整体动作一致率约为 28.0%，其中 discrete 场景下一致率约为 80.3%，但 pixel-goal 场景下的一致率只有约 5.7%。这说明当前这条 replay 基线非常适合用来做“纯推理速度”和“单变量优化收益”的横向比较，但还不能直接用来回答“为什么 realworld 里 System 1 很少发生”这种更深层的系统问题。

基于这组结果，下一步方向反而更加清楚了。短期内，应该继续保留这条 Level 1 replay baseline，把它作为 `num_history`、prompt 长度、`max_new_tokens`、KV cache 和推理后端实验的筛选器，因为它能稳定测出 System 2 的纯推理变化。与此同时，下一条真正值得投入开发的人力，是基于 `replay_v2` 做一个 `decision-only replay benchmark`，只重放那些真实触发新 System 2 决策的节点。只有这样，后面我们才有机会在离线环境里更严格地测量大小脑交接，而不是继续把大量属于同一 pixel-goal 周期的执行步错误地当作新的高层决策来重放。

今天这之后又把 `num_history` 这一组实验重新跑了一遍，而且这次修正了 benchmark 的实现方式，确保它不再复用旧 replay 里已经保存好的历史帧索引，而是真的按 `num_history=8/4/2/0` 重新构造输入。新的结果非常关键，因为它第一次真正回答了“历史帧长度到底是不是 System 2 的主瓶颈之一”。答案是肯定的，而且影响比预期还大。把 `num_history` 从 8 降到 4 之后，离线 replay 的单步时延中位数大约从 0.86 秒降到了 0.48 秒，几乎是直接减半；继续降到 2 时，中位数还能进一步降到约 0.32 秒，降到 0 时甚至可以到约 0.21 秒。与此同时，显存峰值也随之下降，从大约 17.5GB 下降到了 16GB 左右。这说明视觉历史输入的 prefill 成本，确实是当前 System 2 推理链路中最主要、最值得优先优化的部分之一。

但这组实验也同时给出了一个很重要的边界条件。虽然 `num_history` 越小速度越快，但并不是越小越适合直接落回闭环。因为随着历史帧数量下降，模型的输出行为也在变化。`num_history=4` 时，时延几乎减半，但整体一致性没有出现结构性崩塌，而且仍然保留了 `pixel_goal -> System 1` 的路径；但继续压到 `num_history=2` 时，虽然速度更快，离线 replay 中实际触发 System 1 的次数几乎就只剩下 1 次了；压到 `num_history=0` 时，模型几乎完全退化成只走离散动作、不再进入 System 1 的模式。这说明 2 和 0 更像是激进压缩，而不是当前最稳妥的部署候选。

因此，今天新的结论已经比上午清楚得多。此前我们以为也许要立刻去做 KV cache、推理后端之类更重的系统优化，但现在看，更合理的下一步其实是先抓住一个已经被数据明确支持、而且实现成本非常低的优化点，也就是先把 `num_history` 从 8 调到 4，直接放回 Habitat 闭环重跑一次。如果闭环里的 Success、SPL、Navigation Error，以及 `pixel_goal` 比例和每次进入 System 1 后的连续执行步数都没有明显退化，那么这就会是我们第一个可以确认有效的低风险优化。

在这个基础上，KV cache 和推理后端优化当然仍然值得做，而且大概率是下一阶段的重点，但它们的排序应该放在 `num_history=4` 闭环验证之后。原因很简单：现在我们已经知道历史帧长度是明显瓶颈，而把这个变量改掉几乎不需要动模型内部实现；如果这一步就能在闭环里拿到实实在在的收益，就应该先把这个“低成本高确定性”的结果吃到，再去投入更多人力做 KV cache 这类改动更深、调试更复杂的工作。换句话说，现在不是不做 KV cache，而是先把已经找到的便宜收益验证完，再进入更重的系统优化阶段。

为了把这个阶段的执行成本再降下来，今天还额外补了一个一键闭环 runner。这个 runner 可以直接在 Habitat 闭环里批量 sweep `num_history=4/2/1/0`，并且支持两种数据模式：一种是继续使用 mini 子集，另一种是直接使用完整 `val_unseen` 数据，只通过参数决定是否限制 episode 数量。这样后面如果我们要正式验证 `num_history=4` 是否真的适合放回闭环，不需要再手动拷贝配置文件或逐个修改参数，而是可以一条命令把整组实验跑完，并自动输出对比表。这个改动的意义不在于新增了一个算法，而在于把“离线筛选出的候选参数，放回闭环验证”这一步真正工程化了，减少人工操作带来的重复成本和出错风险。

这个一键闭环 runner 在第一次运行时还暴露了一个实现细节问题，也就是新增的 `replay_v2` 路径索引和原来的图像历史缓存不同步，导致程序在构造历史帧输入路径时出现数组越界。这个问题已经在当天修复掉了，本质上属于工程时序上的小 bug，不影响我们前面对实验结论的判断。修复以后，闭环 runner 的用途就明确了：它将成为接下来验证 `num_history=4` 是否能稳定放回 Habitat 闭环的标准入口，而不是一次性的临时脚本。
